        # sqlite — Типы и type affinity

        Homework-шаблон для урока **l2_types_and_affinity** (Типы и type affinity) на платформе Vibe Learn.

        ## Что делать

        На stdlib `sqlite3` реализуй модуль для аудита и нормализации типов. Функции:
classify(value) → возвращает storage class через SELECT typeof(?); audit_column(conn, table, col)
→ словарь {storage_class: count} по реальным классам значений в колонке; find_dirty(conn, table,
col) → строки, где typeof не входит в ожидаемый набор; migrate_to_strict(conn, table, schema) →
создаёт новую STRICT-таблицу, переносит данные с CAST и подменяет старую. Тесты гоняют на
:memory:-базе: проверяют classify на всех 5 классах, что '123' в INTEGER-колонке становится
integer, а 'hello' — text; что STRICT-таблица отвергает несовместимую вставку (ожидается
исключение); и корректность миграции.

Аналитики жалуются: `SELECT ... ORDER BY price` сортирует «как попало», а

## Recap из урока

- В SQLite типы **динамические**: тип принадлежит значению (5 storage classes: NULL/INTEGER/REAL/TEXT/BLOB), а не колонке — в отличие от PostgreSQL.
- У колонки не тип, а **affinity** («склонность»): TEXT/NUMERIC/INTEGER/REAL/BLOB, вычисляется из имени по правилам подстроки (INT→INTEGER, CHAR→TEXT, …).
- Affinity лишь **мягко приводит** значение; не получилось — кладёт как есть. В INTEGER-колонку 'hello' ляжет как TEXT, '123' → INTEGER. Отсюда `'123' != 123`.
- `INTEGER PRIMARY KEY` (именно так) — **псевдоним rowid**, особый случай, не обычная affinity-колонка.
- Нужна строгость как в PG — бери **STRICT-таблицы** (3.37+): `CREATE TABLE ... STRICT` отвергает несовместимые значения с ошибкой.
